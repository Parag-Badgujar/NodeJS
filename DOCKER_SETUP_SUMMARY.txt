Q: Why is docker-compose.yml using "db" as host instead of "localhost"?
A: Inside Docker containers, services communicate by name. "db" is the
   service name defined in docker-compose.yml. Localhost would fail.

Q: How do I access the database from my machine?
A: Use "localhost:5432" with any database client. Docker maps the port.

Q: What if port 4001 is already in use?
A: Change it in docker-compose.yml or kill the process using the port:
   Windows: netstat -ano | findstr :4001
   Linux:   lsof -i :4001

Q: How do I view logs if something goes wrong?
A: docker-compose logs -f [service_name]
   Example: docker-compose logs -f app

Q: Will my database data persist after docker-compose down?
A: Yes! There's a named volume 'postgres_data' for persistence.
   To delete data: docker-compose down -v

Q: How do I update code without rebuilding the image?
A: Development mode uses volume mounts. Changes auto-sync:
   - Edit files locally
   - Changes reflect in container (due to volume: .:/app)
   - App auto-reloads if using nodemon/ts-node-dev

Q: What about production deployment?
A: The Docker image is pushed to GitHub Container Registry.
   It can be deployed to: AWS ECS, Google Cloud Run, Kubernetes, etc.

ISSUE: docker: command not found
FIX:   Install Docker Desktop from https://www.docker.com/products/docker-desktop

ISSUE: Docker daemon not responding
FIX:   Start Docker Desktop application

ISSUE: Port 4001 already in use
FIX:   netstat -ano | findstr :4001 (Windows)
       Then kill process or use different port in docker-compose.yml

ISSUE: App can't connect to database
FIX:   Check .env has DB_HOST=db (NOT localhost)
       Run: docker-compose logs db

ISSUE: Docker image build fails
FIX:   docker system prune -a (clean cache)
       docker-compose up -d --build (rebuild)

ISSUE: Volume mount not working
FIX:   Restart Docker Desktop
       docker-compose down && docker-compose up -d

ISSUE: Can't access container shell
FIX:   docker-compose exec app sh
       Or: docker exec -it CONTAINER_ID bash
